# 行为准则

用于减少 LLM 编码时常见错误。可按需与项目专属说明合并。

权衡：这些准则更偏向谨慎而不是速度。对于非常简单的任务，可自行判断。

## 1. 编码前先思考

不要假设。不要掩盖困惑。明确说明取舍。

实现前：

- 明确说明你的假设；如果不确定，就询问。
- 如果存在多种理解，列出来，不要静默选择。
- 如果有更简单的方案，说明它。必要时提出反对意见。
- 如果需求不清楚，停下来，指出哪里不清楚，并询问。

## 2. 简单优先

用最少代码解决问题。不要做投机式扩展。

- 不添加超出需求的功能。
- 不为单次使用的代码创建抽象。
- 不添加未被要求的“灵活性”或“可配置性”。
- 不为不可能发生的场景添加错误处理。
- 如果写了 200 行但 50 行就能解决，重写并简化。
- 尽量写注释（添加）

自问：资深工程师会不会认为这过度复杂？如果会，就简化。

## 3. 外科手术式修改

只改必须改的地方。只清理自己造成的问题。

编辑现有代码时：

- 不顺手“优化”相邻代码、注释或格式。
- 不重构没有问题的代码。
- 匹配现有风格，即使你会用不同方式实现。
- 如果发现无关的废弃代码，只说明，不删除。

当你的改动造成遗留项时：

- 移除因你的改动而变成未使用的 import、变量或函数。
- 不删除原本就存在的废弃代码，除非用户要求。

检验标准：每一行改动都应能直接追溯到用户请求。

## 4. 目标驱动执行

定义成功标准。持续迭代直到验证通过。

把任务转化为可验证目标：

- “添加校验” → 为非法输入写测试，再让测试通过。
- “修复 bug” → 写一个复现 bug 的测试，再让测试通过。
- “重构 X” → 确保重构前后测试都通过。

对于多步骤任务，给出简短计划：

1. [步骤] → 验证：[检查项]
2. [步骤] → 验证：[检查项]
3. [步骤] → 验证：[检查项]

清晰的成功标准可以让你独立循环推进。模糊标准（如“让它能工作”）需要持续澄清。

## 5.附加
- 注释只解释不明显的业务规则或复杂逻辑。
- 避免过度抽象，避免使用func作为参数
- 避免使用过长的方法调用结果作为参数传入，影响可读性
- 代码要保证易读性，方法调用时需要添加注释说明

## 6. 维护 CHANGELOG

每次修改代码或项目文档时，同步更新根目录的 `CHANGELOG.md`，让本次改动可回溯、可复制为提交信息。

记录规则：

- 每次任务至少新增一条记录；如果只是纯检查且没有改动文件，可以不写。
- `CHANGELOG.md` 按功能点分组，不按日期分组；二级标题使用功能点名称，例如 `## Agent 行为准则`、`## 小红书线索挖掘`。
- 新记录先查找是否已有对应功能点；如果没有，再新增功能点标题。
- 同一功能点下，时间相近、目标一致的多次改动应合并到同一条记录中。
- 同一功能点下，如果改动时间相隔较远或上下文已经不同，不要强行合并；新增独立条目，并在条目末尾标注日期。
- 条目使用 Git Commit 风格，优先采用 `type(scope): summary`。
- `type` 优先使用 `feat`、`fix`、`docs`、`refactor`、`test`、`chore`。
- `scope` 使用被修改的模块、目录或功能名，例如 `agents`、`client`、`server`、`xhs`。
- `summary` 用一句话说明实际改动，避免泛泛写“更新代码”。
- 如有必要，在条目下用 1-3 个子项说明关键细节；不要写实现流水账。

示例：

```text
## Agent 行为准则

- docs(agents): 添加 CHANGELOG 维护规则（2026-05-24）
  - 约定每次代码或文档改动后同步记录 commit-style 变更摘要。
  - 后续相近改动合并到同一功能点，避免按日期拆散上下文。
```

如果这些准则有效，结果应是：diff 中不必要的改动更少，因过度复杂导致的返工更少，澄清问题发生在实现之前而不是出错之后。
