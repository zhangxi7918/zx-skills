# 代码审查技能

## 触发方式

当用户请求代码审查、review 代码、检查代码质量，或显式输入 `/review` 时使用本 skill。

## 执行步骤

1. **分析代码结构**
   - 识别函数、类、模块和组件的职责。
   - 理解关键数据流、控制流、状态变化和外部依赖。
   - 优先查看与用户请求、变更 diff、失败测试或风险路径直接相关的代码。

2. **检查项目清单**
   - [ ] 安全隐患：SQL 注入、XSS、硬编码密钥、权限绕过、敏感信息泄露、不可信输入处理。
   - [ ] 性能问题：循环嵌套、重复查询、不必要的重绘、过度渲染、阻塞 IO、缓存缺失。
   - [ ] 可读性：命名、注释、复杂度、重复逻辑、过长函数、职责混杂。
   - [ ] 最佳实践：错误处理、边界条件、资源释放、并发一致性、类型约束、框架约定。
   - [ ] 测试覆盖：关键路径、边界情况、失败路径、回归风险、缺失断言。

3. **生成报告**
   - 按严重程度排序问题。
   - 每条问题必须给出具体位置，优先使用 `文件:行号`。
   - 每条问题必须说明风险或影响，并给出可执行的修复建议。
   - 如果没有发现明确问题，直接说明未发现问题，并列出剩余测试缺口或残余风险。
   - 给出整体评分，范围为 1-10。

## 输出格式

```markdown
## 代码审查报告

### 严重问题 🔴
- [问题描述] at `文件:行号`
  - 风险/影响：XXX
  - 修复建议：XXX

### 一般问题 🟡
- [问题描述] at `文件:行号`
  - 风险/影响：XXX
  - 修复建议：XXX

### 建议与改进 🟢
- [建议描述] at `文件:行号`
  - 改进建议：XXX

### 测试缺口
- XXX

### 总体评分：X/10
```

## 审查原则

- 结论优先，先列问题，再补充背景。
- 不因为代码风格偏好输出问题；只报告会造成缺陷、维护成本、性能风险、安全风险或测试盲区的事项。
- 严重问题用于可能导致数据错误、安全漏洞、崩溃、不可恢复状态或重大用户影响的缺陷。
- 一般问题用于局部行为风险、可维护性问题、性能浪费或边界条件遗漏。
- 建议与改进用于低风险优化，必须保持具体、可执行。
- 不确定时明确说明假设，并指出需要补充验证的证据。
