# 面试复盘

## 适用范围

用于求职 workspace 的面试复盘任务。分析时结合当前 workspace 的简历和项目代码背景：

- `简历/`
- `ai_customer_mining/`
- `ASagent/`

不要维护旧的长期面试复盘 hub。后续只创建单次复盘文档，并在公司状态变化时更新求职跟踪表。

## 输入处理

1. 如果用户提供飞书妙记链接，优先读取妙记总结、章节、待办和逐字稿。
2. 如果用户提供录音或 transcript，只取完成分析所需内容。
3. 不生成或保留本地 transcript 文件，除非用户明确要求。
4. 不生成本地 review markdown 文件，除非用户明确要求。
5. 信息缺失时，先根据已有内容做有限判断；只有公司、岗位、轮次、文档写入位置这类关键字段无法推断时才询问。

## 单次复盘文档

在 workspace `AGENTS.md` 指定的飞书 wiki folder 下创建新文档。

推荐标题：

```text
YYYY-MM-DD_公司_岗位或方向_面试复盘
```

文档建议结构：

```markdown
# YYYY-MM-DD 公司 岗位/方向 面试复盘

## 基本信息

- 公司：
- 岗位/方向：
- 面试轮次：
- 面试日期：
- 结果判断：
- 核心风险：
- 下一步：

## 总体判断

## 核心问题复盘

### 面试问题本身？

- 问题
- 原回答要点
- 原回答逐字稿
  > 引用原话
- 评价
- 推荐回答
  > 推荐回答文本

## 项目真实性与技术深度

## 后续准备重点
```

`核心问题复盘` 的每个 subsection 标题必须写成面试官问题或技术问题本身，例如 `React 使用经验怎么展开？`，不要写成 `React 使用经验回答太薄` 这类评价句。

## 分析要求

- 判断项目回答时，要对照简历和两个项目代码库，区分“真实做过但没讲清楚”和“确实缺少技术深度”。
- 推荐回答要能直接用于下次面试，尽量用第一人称。
- 对薄弱点给出可执行补救方向，不要只写抽象建议。
- 可以保留必要逐字稿引用，但不要把整段 transcript 变成归档。
- 结论要直接，不要为了安慰弱化风险。

## 求职跟踪表

当 interview/company status changed，或用户明确要求记录/同步时，更新 workspace `AGENTS.md` 指定的求职跟踪表。

优先使用工作表：

```text
求职跟踪（简化版）
```

保持行轻量，字段按这个顺序维护：

- 公司
- 岗位
- 来源/联系人
- 当前阶段
- 下一步
- 日期/反馈
- 状态
- 复盘/录音
- 备注

把单次复盘文档链接放入 `复盘/录音`。

## 完成前检查

- 单次复盘文档已创建并可访问。
- 文档包含 `基本信息`、`总体判断`、`核心问题复盘`、`后续准备重点`。
- 如更新了求职跟踪表，确认对应行已写入或更新。
- 没有遗留本地 transcript 或 review markdown 文件，除非用户要求。

最终回复只需要给出复盘文档 URL、是否更新求职跟踪表、以及简短验证结果。
