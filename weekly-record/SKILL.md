# 周刊记录

默认文档：
https://my.feishu.cn/wiki/Hc8MwJLkOi7naLkHj9bcu0NTnMh

## 风格

- 每周一个 `h2`：`YYYY.MM.DD~YYYY.MM.DD`，周一到周日。
- 新周在上，旧周在下。
- 周标题下只放有序列表。
- 每条是“看到什么 + 为什么对我有用 / 我打算怎么用”。
- 用用户自己的口吻，短、直接、口语化；不要写周报腔、总结腔。
- 可以保留链接、附件、图片；不要为了美观重排整篇文档。
- 讨论时留意可延展成博客的内容：有反复出现的问题、清晰的方法论、可复盘项目、强观点或适合调研的主题。

## 流程

1. 先判断用户意图：
   - 用户发资料、想法、问题，或说“聊聊/怎么看/探讨一下”时，先讨论、追问、给建议，不急着写入。
   - 用户明确说“记录/写入/更新到周刊/记到文档里”时，也要先给待写入草稿，等用户确认后再更新文档。
2. 如果是链接、概念、工具、文章，可以先调研或讨论，再提炼成条目。
3. 用户记录的问题如果有更好的角度、行动建议或风险提醒，先直接说出来，和用户确认要不要写进周刊。
4. 如果素材适合后续写博客，顺手提示“可写博客”，给 1 个具体选题角度；不要每条都硬凑。
5. 每条压成 1 句为主，必要时加 1-2 个子 bullet。不要把资料摘要塞成长文。
6. 用 Asia/Shanghai 当前日期计算本周标题。
7. 使用 `lark-doc` 读取目录：
   ```bash
   lark-cli docs +fetch --api-version v2 --doc "<默认文档>" --scope outline --max-depth 2 --detail with-ids
   ```
8. 如果本周标题存在，读取该 section，找到有序列表：
   - 有空的 `<li></li>` 时，用第一条替换它；
   - 已有内容时，在最后一个顶层 `<li>` 后插入新 `<ol><li seq="auto">...</li></ol>`。
9. 如果本周标题不存在，读取文档开头，找到说明段“记录 看到了什么...”，在它后面插入：
   ```xml
   <h2>YYYY.MM.DD~YYYY.MM.DD</h2>
   <ol>
     <li seq="auto">第一条</li>
     <li seq="auto">第二条</li>
   </ol>
   <p></p>
   ```
10. 更新飞书前，必须先把准备写入的条目发给用户确认。
11. 用户确认后，再用 `docs +update --api-version v2` 更新；优先 `block_insert_after` / `block_replace`，不要 `overwrite`。

## 写法

好：
- `Agent Browser是给Agent用的CLI工具。可以去了解一下CLI是怎么做出来给Agent用的。`
- `新闻、资讯获取经常会发散，是否可以把这类活动SOP化。`

不好：
- `本周我深入学习了若干前沿技术，并对未来发展趋势形成了系统性认知。`
- `该工具显著提升效率，值得持续关注。`

## 完成前

- 简短告诉用户新增了哪几条、写进了哪个周区间。
- 如果发现适合写博客的内容，单独列出“可延展选题”和推荐角度。
- 任何写入飞书前，都必须先给用户看草稿并等待确认。
- 如果用户只是让你“整理一下”或“探讨一下”，先给观点和草稿，不更新文档。
